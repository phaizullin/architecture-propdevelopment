# Проверочный лист по безопасности для бизнес-систем PropDevelopment

На основе анализа вводных данных (включая проблемы с контролем данных, API партнеров и соответствием законодательству) и стандартов ISO/IEC 27001/27002, я выбрал следующие разделы для checklist: классификация данных, контроль доступа, безопасность интеграций, хранение и резервное копирование, мониторинг и аудит, управление рисками и инцидентами, соответствие требованиям. Выбор обоснован приоритетами компании - фокус на предотвращении утечек, обеспечении конфиденциальности и интеграционной безопасности, как указано в вводных (например, нарушения контроля данных и API риски).

## Классификация данных

| Контроль/Вопрос | Статус (Да/Нет/Не применимо) | Рекомендация/Комментарий |
|-----------------|------------------------------|---------------------------|
| Все данные классифицированы по уровням (публичные, внутренние, конфиденциальные, секретные)? | | Обновить классификацию на основе mindmap из Task1; внедрить политики маркировки данных. |
| Есть ли политики обработки и хранения для каждой категории данных? | | Разработать и задокументировать политики в соответствии с ISO 27002 (раздел 8). |
| Проводится ли регулярный обзор классификации данных? | | Внедрить ежегодный аудит классификации для учета изменений в бизнес-процессах. |

## Контроль доступа

| Контроль/Вопрос | Статус (Да/Нет/Не применимо) | Рекомендация/Комментарий |
|-----------------|------------------------------|---------------------------|
| Используется ли многофакторная аутентификация (MFA) для доступа к конфиденциальным и секретным данным? | | Обязательно внедрить MFA в системах вроде client-crm-app и tenant-core-app (ISO 27001 A.9). |
| Определены ли роли и разрешения (RBAC) для пользователей (менеджеры, аналитики, клиенты)? | | Внедрить RBAC, чтобы предотвратить несанкционированный доступ, особенно в CRM и финансовых системах. |
| Проводится ли регулярный аудит логов доступа и отзыв устаревших прав? | | Настроить автоматизированный аудит и ежеквартальный отзыв прав для минимизации рисков. |

## Безопасность интеграций

| Контроль/Вопрос | Статус (Да/Нет/Не применимо) | Рекомендация/Комментарий |
|-----------------|------------------------------|---------------------------|
| API партнеров (ЖКХ, платежные системы) проверены на наличие уязвимостей и утечек? | | Провести аудит API контрактов, добавить scoping для предотвращения доступа к чужим данным (как в вводных). |
| Используется ли шифрование (TLS/HTTPS) для всех внешних передач данных? | | Обязательно внедрить TLS 1.3 для всех интеграций, включая СМЭВ и партнеров. |
| Есть ли мониторинг и rate limiting для предотвращения атак через интеграции? | | Внедрить WAF и мониторинг для API, чтобы избежать проблем, описанных в вводных. |

## Хранение и резервное копирование

| Контроль/Вопрос | Статус (Да/Нет/Не применимо) | Рекомендация/Комментарий |
|-----------------|------------------------------|---------------------------|
| Базы данных (client-mart-db, tenant-core-db и т.д.) имеют регулярное резервное копирование? | | Убедиться в соответствии с требованиями к глубине бэкапов, протестировать восстановление (ISO 27001 A.12). |
| Данные шифруются на хранении (encryption at rest)? | | Внедрить шифрование для конфиденциальных и секретных данных в БД. |
| Есть ли планы восстановления после инцидентов (disaster recovery)? | | Разработать и протестировать DR-планы для ключевых систем. |

## Мониторинг и аудит

| Контроль/Вопрос | Статус (Да/Нет/Не применимо) | Рекомендация/Комментарий |
|-----------------|------------------------------|---------------------------|
| Ведется ли логирование всех доступов и изменений данных? | | Интегрировать централизованное логирование (SIEM) для всех систем. |
| Есть ли система обнаружения инцидентов (IDS/IPS)? | | Внедрить SIEM для реального времени мониторинга, особенно для интеграций. |
| Проводятся ли регулярные аудиты на соответствие ФЗ-152 и GDPR? | | Запланировать ежегодные аудиты для соответствия законодательству РФ. |

## Управление рисками и инцидентами

| Контроль/Вопрос | Статус (Да/Нет/Не применимо) | Рекомендация/Комментарий |
|-----------------|------------------------------|---------------------------|
| Идентифицированы ли риски для каждой категории данных (из mindmap)? | | Связать с mindmap Task1, обновить реестр рисков. |
| Есть ли план реагирования на инциденты (incident response plan)? | | Разработать план, включая уведомление о утечках в 72 часа, как у конкурента в вводных. |
| Оценены ли риски по шкале (незначительный/значительный/критический)? | | Проводить регулярную оценку рисков на основе mindmap. |

## Соответствие требованиям

| Контроль/Вопрос | Статус (Да/Нет/Не применимо) | Рекомендация/Комментарий |
|-----------------|------------------------------|---------------------------|
| Системы соответствуют ФЗ-152 (персональные данные) и другим нормам РФ? | | Провести внешний аудит для подтверждения соответствия. |
| Проверены ли интеграции с госорганами (СМЭВ) на безопасность? | | Обеспечить шифрование и аудит для интеграций с госорганами. |
| Есть ли политики для обработки жалоб клиентов на ошибки данных? | | Внедрить процесс обработки жалоб, как упомянуто в вводных (ошибки в ЛК). |
