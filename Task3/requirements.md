# Требования к внешним интеграциям для сервисов Умный дом

Требования к безопасности и интеграции для новых сервисов "Умный дом" в системе PropDevelopment:
- **Интеллектуальный домофон** (видеозвонки, биометрическое распознавание, удаленное управление)
- **Интеллектуальный шлагбаум** (распознавание номеров автомобилей, автоматическое управление)

## 1. Требования к безопасности

### 1.1. Шифрование и защита данных
- **Транспортное шифрование**: Обязательное использование TLS 1.3 для всех внешних API
- **Шифрование данных в покое**: Биометрические данные и изображения должны шифроваться с использованием AES-256
- **Хеширование**: Биометрические шаблоны должны храниться в хешированном виде (SHA-256 + salt)
- **Защита персональные данных**: Персональные данные (изображения лиц, номера авто) должны быть псевдонимизированы

### 1.2. Сетевая безопасность
- **Firewall**: Разрешить доступ только к необходимым портам и IP-адресам
- **VPN туннели**: Использовать VPN для связи с критически важными системами партнеров
- **Rate limiting**: Ограничить количество запросов (100 запросов/минуту на клиента)
- **DDoS защита**: Внедрить защиту от DDoS атак на уровне API Gateway

### 1.3. Мониторинг и аудит
- **Логирование**: Все API вызовы должны логироваться с включением timestamp, source IP, user ID
- **SIEM интеграция**: Логи должны поступать в централизованную SIEM систему
- **Алерты**: Настроить уведомления о подозрительной активности (множественные неудачные попытки доступа)
- **Журнал событий IoT**: Вести детальный журнал всех команд, отправленных на устройства

## 2. Протоколы аутентификации и авторизации

### 2.1. Аутентификация пользователей
```
Схема: OAuth 2.0 + JWT
- Authorization Code Flow для мобильного приложения
- Client Credentials Flow для server-to-server интеграций
- Время жизни токенов: Access Token (15 мин), Refresh Token (30 дней)
- Сlaims в JWT: user_id, apartment_id, permissions, exp, iat
```

### 2.2. Авторизация доступа к устройствам
```
RBAC модель устройств:
- Роль "Owner": полный доступ ко всем устройствам квартиры
- Роль "Resident": доступ к домофону, ограниченный доступ к шлагбауму
- Роль "Guest": временный доступ только к домофону
- Роль "Service": доступ для технического обслуживания
```

### 2.3. Аутентификация внешних систем
```
API Key Authentication:
- Уникальные API ключи для каждого партнера
- Ротация ключей каждые 90 дней
- Подпись запросов с использованием HMAC-SHA256
- Header: Authorization: Bearer <api_key>
- Header: X-Signature: <hmac_signature>
```

### 2.4. Биометрическая аутентификация
```
Протокол:
1. Захват биометрических данных (лицо/номер авто)
2. Локальная предобработка и создание шаблона
3. Отправка зашифрованного шаблона в систему распознавания
4. Сравнение с базой разрешенных шаблонов
5. Возврат решения (разрешить/запретить) без передачи исходных данных
```

## 3. Организация взаимодействия между системами

### 3.1. Архитектура взаимодействия

```
Мобильное приложение
    ↓ HTTPS + JWT
API Gateway (Kong/Nginx)
    ↓ REST API
smart-home-service
    ↓ REST API + API Keys
integration-service
    ↓ TLS 1.3 + OAuth 2.0
Внешние системы партнеров
    ↓ MQTT + TLS
IoT устройства
```

### 3.2. Протоколы связи

#### 3.2.1. PropDevelopment ↔ Платформа Умный дом
```
Протокол: REST API over HTTPS
Аутентификация: API Key + OAuth 2.0
Формат: JSON
```

#### 3.2.2. PropDevelopment ↔ Биометрическая система
```
Протокол: REST API over HTTPS
Аутентификация: Bearer Token (JWT)
Формат: JSON + Base64 encoded images
```

#### 3.2.3. Платформа Умный дом ↔ IoT устройства
```
Протокол: MQTT over TLS
```

### 3.3. Обработка ошибок и отказоустойчивость

#### 3.3.1. Circuit Breaker Pattern
```
Параметры:
- Threshold: 5 неудачных запросов подряд
- Timeout: 30 секунд
- Recovery: Exponential backoff (1s, 2s, 4s, 8s, max 60s)
```

#### 3.3.2. Retry Policy
```
Стратегия:
- Количество попыток: 3
- Backoff: Exponential (1s, 2s, 4s)
- Retry коды: 502, 503, 504, timeout
- No retry коды: 400, 401, 403, 404
```

#### 3.3.3. Fallback механизмы
```
Для домофона:
- При отказе биометрии → стандартный видеозвонок
- При отказе API → локальное управление

Для шлагбаума:
- При отказе распознавания → ручное управление через приложение
- При отказе связи → аварийный режим (остается закрытым)
```

## 4. Требования к данным

### 4.1. Классификация данных
```
Биометрические данные: КРИТИЧЕСКИЙ уровень
- Изображения лиц
- Биометрические шаблоны
- Номера автомобилей

Операционные данные: КОНФИДЕНЦИАЛЬНЫЙ уровень
- Логи доступа
- Настройки устройств
- Временные доступы

Технические данные: ВНУТРЕННИЙ уровень
- Статус устройств
- Метрики производительности
```

### 4.2. Хранение и ретенция
```
Биометрические шаблоны: шифрование AES-256, хранение 5 лет
Изображения (для аудита): шифрование AES-256, хранение 30 дней
Логи событий: хранение 1 год
Статистика использования: анонимизация, хранение 3 года
```
